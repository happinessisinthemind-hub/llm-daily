# Slack AI Relay / Dispatcher 運用ルール

この文書は、Slack上で複数のAIエージェントを連携させるときの「中継ルール」をまとめたものです。

AI同士を直接会話させることだけに頼らず、誰が、誰に、何を、どこまで依頼するのかを明確にして、うにゃんにゃん保健室のAI編集部を安全に動かすために使います。

---

## 1. この文書の目的

Slackには、OpenClaw、Hermes Agent、Codex、Claude Code、Cursor、Grokなど、複数のAIエージェントが参加しています。

ただし、AIエージェント同士が直接 `@Hermes` や `@Grok` のようにメンションしても、受け取る側が反応しないことがあります。そこで、AI同士の直接会話に依存せず、中央に「AI Relay / AI Dispatcher」という考え方を置きます。

この文書の目的は次の4つです。

- Slack上で複数AIエージェントを安全に連携させる
- bot投稿に受信側AIが反応しない問題を回避する
- AI同士の無限ループや同じ依頼の繰り返しを防ぐ
- 最終判断は必ず人間が行う形にする

ここでいう **AI Dispatcher** は、特定のアプリ名ではなく「依頼先、目的、ループ回数、停止条件を管理する運用上の役割」です。

---

## 2. 現在の問題

現時点では、Slack上で次のような問題が起きることがあります。

- AIエージェントから別のAIエージェントへ `@メンション` しても、相手が反応しないことがある
- 同じ内容を人間がコピーして投稿すると、相手のAIが反応することがある
- 受信側AIが、botからの投稿を無視している可能性がある
- Slack側の設定やアプリ側の仕様が関係している可能性がある

考えられる原因は、ひとつに決めつけないようにします。たとえば次のような要素が関係しているかもしれません。

| 観点 | ありえる原因 |
| --- | --- |
| メンション形式 | AIが投稿したメンションが、受信側アプリに正しく届いていない |
| チャンネル参加 | 受信側AIがそのチャンネルに参加していない |
| Slack権限 | アプリのスコープやイベント購読が足りない |
| bot投稿フィルター | 受信側AIがbot投稿を意図的に無視している |
| スレッド仕様 | スレッド内の投稿や返信を受信側AIが拾えていない |

そのため、「AIがAIを呼べば必ず動く」という前提は置かないことにします。

---

## 3. 基本方針

AI Relay / Dispatcher運用では、次の方針を基本にします。

- AI同士を直接 `@メンション` で会話させることに依存しない
- 依頼内容は、できるだけ **AI_CALL形式** の依頼書として書く
- AI Dispatcherが、依頼先・目的・ループ回数・停止条件を管理する
- AI同士のやり取りは最大3回までにする
- 3回で解決しない場合は、人間確認に切り替える
- 外部投稿、公開、送信、PRマージ、削除は必ず人間承認にする

大事なのは、AIに自由に会話を続けさせることではありません。

「どのAIに、何を見てもらい、どんな形で返してもらい、どこで止めるか」を先に決めてから渡すことです。

---

## 4. AI_CALL形式

AI同士の受け渡しには、次のテンプレートを使います。

```text
[AI_CALL]
target:
from:
loop_id:
round:
task:
context:
output_format:
reply_required:
stop_after_reply:
human_approval_required:
```

### 各項目の意味

| 項目 | 書く内容 |
| --- | --- |
| `target` | 依頼先のAI名。例: Hermes Agent、Claude Code、Codex |
| `from` | 依頼元。例: OpenClaw、うーちゃん、Codex |
| `loop_id` | ひとまとまりの作業ID。例: `note-article-idea-2026-06-24` |
| `round` | 何回目のやり取りか。最大3回まで |
| `task` | 依頼したい作業を短く書く |
| `context` | 背景、前提、参照してほしい情報 |
| `output_format` | 返答形式。例: 箇条書き、表、Markdown、TODOリスト |
| `reply_required` | 返信が必要か。例: `yes` / `no` |
| `stop_after_reply` | 返信後に止めるか。例: `yes` / `no` |
| `human_approval_required` | 人間承認が必要か。例: `yes` / `no` |

### 記入例

```text
[AI_CALL]
target: Hermes Agent
from: OpenClaw
loop_id: note-next-idea-001
round: 1/3
task: 既存noteの内容を要約し、次の記事案に使える論点を整理してください。
context: うにゃんにゃん保健室のnote企画。読者は医療・介護・心理に関心がある一般の人。断定表現は避ける。
output_format: 要約、決定事項、TODO、不明点、改善点の箇条書き
reply_required: yes
stop_after_reply: yes
human_approval_required: yes
```

`stop_after_reply: yes` の場合、受け取ったAIは返信したらそこで止まります。次のAIに勝手に依頼を回さないようにします。

## 5. 役割分担

Slack上のAIエージェントは、次のように役割を分けます。

| 担当 | 役割 | 主な仕事 |
| --- | --- | --- |
| OpenClaw | 司令塔 | 作業分解、担当AIの選定、次工程の決定 |
| Hermes Agent | 記録係 | 要約、決定事項、TODO、不明点、改善点の整理 |
| Claude Code | レビュー係 | 文章の自然さ、構成、読者目線、安全性の確認 |
| Codex | 成果物化係 | Markdown保存、README修正、PR作成 |
| Cursor | 現場修正係 | ローカル編集、細かい調整、実装補助 |
| Grok | 別視点係 | アイデア出し、切り口確認、意見の対比 |
| 人間 | 最終承認者 | 公開、送信、マージ、削除、採用判断 |

AIに役割を持たせることで、「全員が同じことを言う」「同じ依頼が何度も回る」「誰が止めるのか分からない」という状態を避けます。

## 6. Slackチャンネル設計

チャンネルは、目的ごとに分けると混乱しにくくなります。

| チャンネル例 | 使い方 |
| --- | --- |
| `#ai司令室` | 依頼の入口。OpenClawへの相談、AI Dispatcherの判断、全体方針の整理 |
| `#agent-relay-test` | AI_CALL形式やbot反応の実験用チャンネル |
| `#note企画` | うにゃんにゃん保健室のnote企画、記事案、構成案の相談 |
| `#ai作業ログ` | AIが行った作業、決定事項、未完了TODOの記録 |
| `#成果物` | Markdown、記事案、README修正案、PRなどの成果物置き場 |
| `#決定事項` | 人間が確認して決めたことだけを残す場所 |

最初からすべてを厳密に分ける必要はありません。まずは `#agent-relay-test` で試し、うまく回りそうな流れだけ本番チャンネルへ移すのがおすすめです。

## 7. 安全ルール

AI Relay / Dispatcher運用では、安全のために次のルールを守ります。

### ループ防止

- bot同士のループは最大3回までにする
- 同じ依頼を繰り返さない
- `loop_id` と `round` を見て、今が何回目か分かるようにする
- 3回でまとまらない場合は、人間確認に切り替える

### 投稿とメンション

- 送信元botを許可リスト化し、想定外のbotからの依頼は自動で進めない
- `@channel`、`@here`、`@everyone` はAIが勝手に使わない
- AIだけで外部公開や投稿をしない
- Slack内でも、重要な告知や広範囲メンションは人間が確認してから行う

### 内容の安全性

- 医療・介護・心理・法務・契約・請求に関する断定は避ける
- 個人情報や機密情報を含む場合は、人間確認に回す
- 請求、契約、支払い、削除、公開、送信、PRマージは人間承認にする
- 迷ったら止めて、人間確認にする

この運用では、「AIが止まれること」も大事な機能として扱います。

## 8. うにゃんにゃん保健室 note運用での使用例

ここでは、「既存noteを読んで、次の記事案を作る」場合の流れを例にします。

```text
うーちゃんが依頼
↓
OpenClawが作業分解
↓
Hermesが要約とTODO整理
↓
Claude Codeがnote案をレビュー
↓
CodexがMarkdownで保存
↓
うーちゃんが確認
```

### 具体的な流れ

1. うーちゃんが `#note企画` または `#ai司令室` に依頼を書く
2. OpenClawが目的、読者、必要な作業を分解する
3. OpenClawがHermes Agent向けのAI_CALLを作る
4. Hermes Agentが、既存noteの要約、決定事項、TODO、不明点を整理する
5. OpenClawまたは人間が、Claude Code向けのAI_CALLを作る
6. Claude Codeが、文章の自然さ、構成、読者目線、安全性をレビューする
7. Codexが、決まった内容をMarkdownとして保存し、必要ならPRを作成する
8. うーちゃんが内容とPRを確認し、採用・修正・マージを判断する

### note運用用のAI_CALL例

```text
[AI_CALL]
target: Claude Code
from: OpenClaw
loop_id: unyannyan-note-next-article-001
round: 2/3
task: Hermes Agentが整理した内容をもとに、次のnote記事案をレビューしてください。
context: うにゃんにゃん保健室の読者向け。やさしい日本語で、医療・介護・心理に関する断定は避けたいです。
output_format: 良い点、気になる点、修正提案、人間確認が必要な点
reply_required: yes
stop_after_reply: yes
human_approval_required: yes
```

この時点でClaude Codeが別のAIへ勝手に依頼を回す必要はありません。次に誰へ渡すかは、OpenClawまたは人間が判断します。

## 9. 実験手順

最初は本番運用にせず、テスト用チャンネルで小さく試します。

1. `#agent-relay-test` を作る
2. 関係するAIエージェントを招待する
3. 人間が最初の指示を出す
4. OpenClawにAI_CALL形式で依頼を作らせる
5. Hermes Agentが反応するか確認する
6. 反応しない場合は、人間がAI_CALLをコピーして投稿する運用に切り替える
7. 何が反応し、何が反応しなかったかを `#ai作業ログ` に残す
8. 将来的には、API経由でAI Dispatcherが投稿を中継する方式を検討する

### 確認ポイント

- AIがbot投稿に反応するか
- スレッド内でも反応するか
- チャンネル投稿なら反応するか
- `AI_CALL` の形式だと依頼内容が伝わりやすいか
- ループ回数と停止条件が守られているか
- 人間確認が必要な作業で止まれているか

反応しないAIがあっても、失敗ではありません。その場合は「人間がAI_CALLをコピーして投稿する」という手動中継を正式な逃げ道として使います。

---

## 10. 最終的なゴール

最終的なゴールは、AI同士を完全自動で会話させることではありません。

目指すのは、うーちゃんが毎回すべてをコピーしなくても、AI同士の受け渡しが分かりやすく整理される状態です。

そのうえで、次の方針は変えません。

- 公開、送信、PRマージ、削除などの重要操作は必ず人間が確認する
- AIは下書き、整理、レビュー、成果物化を助ける
- 最終的に何を採用するかは人間が決める
- Slackを「うにゃんにゃん保健室 AI編集部」として使えるようにする

AI Relay / Dispatcherは、AIを勝手に走らせるための仕組みではなく、AIを安全に止めながら活用するための運用ルールです。
