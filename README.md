## SlackからCodexに依頼するときの基本形

```text
対象リポジトリ: happinessisinthemind-hub/llm-daily
目的: 何を達成したいかを簡潔に書く
やってほしいこと: Codexに実施してほしい作業を箇条書きで書く
触ってよいファイル: 変更してよいファイルやディレクトリを書く
触らないでほしいファイル: 変更してほしくないファイルやディレクトリを書く
完了条件: どうなれば完了か、確認方法やテスト内容を書く
変更後は差分を見せてください
```

## Codex運用時の安全ルール

- 変更前に目的を明確にする
- 触ってよいファイルを指定する
- 秘密情報、APIキー、個人情報、医療・介護の機微情報はSlackに書かない
- 変更後は必ずdiffを確認する
- PRを確認してからマージする
- 大きな変更は小さなタスクに分ける

## Slack運用の役割分担

- Slack: 相談、指示、作業ログを残す場所
- Codex: GitHub上のファイル確認、修正、差分作成、PR作成を担当
- User: 指示内容の決定、diff確認、PR確認、マージ判断を担当
- GitHub: 変更履歴、PR、最終的なファイル管理を行う場所

## AIエージェント利用の優先順位と使い分けルール

### 基本方針

- 日常的なGitHub作業、README修正、diff確認、PR作成は、原則としてCodexを優先する
- CodexはChatGPT Proプラン内で使いやすいため、通常作業の第一候補とする
- Claude CodeやCursorは、Codexでは対応しにくい作業や、別視点のレビューが必要な場合に限定する

### Codexを優先する作業

- README.mdやMarkdownファイルの修正
- 小さなファイル修正
- 差分作成
- PR作成
- GitHub上の構成確認
- Slackからの通常タスク依頼

### Claude Codeに回す作業

- 文章構成や設計の見直し
- 丁寧なレビューや改善提案
- READMEや運用ルールの整理
- Codexの出力を別視点で確認したい場合

### Cursorに回す作業

- Cloud Agentとしてコード全体の確認が必要な場合
- 複数ファイルにまたがるコード修正の補助
- Cursorの開発環境やエディタ機能と組み合わせたい場合

### 節約ルール

- まずCodexに依頼する
- Codexで難しい、または不安が残る場合だけClaude CodeかCursorに依頼する
- 同じ作業を複数AIに同時依頼しない
- Claude CodeやCursorを使う前に「なぜCodexではなくこちらを使うのか」を一言メモする
- 大きな作業は小さく分けて、無駄な利用を避ける

### 共通ルール

- 変更前に対象リポジトリ、目的、触ってよいファイルを明記する
- 変更後は必ずdiffとPRを確認してからマージする
- 秘密情報、APIキー、個人情報、医療・介護の機微情報はSlackに書かない

## AI役割分担ルール

Slack上で複数のAIを使うときは、まず「誰に何を任せるか」を簡単に決めてから依頼します。迷ったときは、最初にCodexへ投げるのを基本にします。Codexで足りない部分が見えた場合だけ、Claude CodeやCursorに回します。

- OpenClawは司令塔です。次の工程、優先順位、どのAIに依頼するかを決めます。
- Codexは実作業の第一候補です。GitHub上のREADME修正、Markdown整理、ファイル編集、コミット、PR作成などを担当します。
- Claude Codeは、長文レビュー、文章の自然さチェック、構成整理、複雑な意図の読み取りが必要なときに使います。
- Cursorは、ローカル編集、画面確認、細かいコード修正、ファイル単位の調整が必要なときに使います。
- Hermes Agentは、会話ログ、決定事項、TODO、日報など、あとで見返す情報の整理を担当します。

Slackで「次」とだけ言われた場合は、いきなり複数の作業を出さず、まず現在の状況を短く整理します。そのうえで、次にやるべき作業を1つだけ提案します。
