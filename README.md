## このリポジトリの使い方

### 基本的な閲覧方法

1. **GitHub上で直接閲覧** - ブラウザでこのリポジトリを開き、各Markdownファイルを読む
2. **ローカルで閲覧** - クローンして任意のMarkdownエディタで開く

```bash
git clone https://github.com/happinessisinthemind-hub/llm-daily.git
cd llm-daily
```

### ドキュメントの更新方法

このリポジトリの更新は、主にSlack経由でAIエージェント（Codex、Claude Code、Cursor）に依頼する運用を想定しています。

**推奨フロー:**

1. Slackで依頼内容を整理（[基本形](#slackからcodexに依頼するときの基本形)を参照）
2. AIエージェントがファイルを修正し、PRを作成
3. diffを確認してからマージ

**直接編集する場合:**

```bash
# 変更を加える
git add .
git commit -m "変更内容の簡潔な説明"
git push origin main
```

## SlackからCodexに依頼するときの基本形

```text
対象リポジトリ: happinessisinthemind-hub/llm-daily
```

## Slack運用の役割分担

- Slack: 相談、指示、作業ログを残す場所
- Codex: GitHub上のファイル確認、修正、差分作成、PR作成を担当
- User: 指示内容の決定、diff確認、PR確認、マージ判断を担当
- GitHub: 変更履歴、PR、最終的なファイル管理を行う場所

## AIエージェント利用の優先順位と使い分けルール

### 基本方針

- 日常的なGitHub作業、README修正、diff確認、PR作成は、原則としてCodexを優先する
- CodexはChatGPT Proプラン内で使いやすいため、通常作業の第一候補とする
- Claude CodeやCursorは、Codexでは対応しにくい作業や、別視点のレビューが必要な場合に限定する

### Codexを優先する作業

- README.mdやMarkdownファイルの修正
- 小さなファイル修正
- 差分作成
- PR作成
- GitHub上の構成確認
- Slackからの通常タスク依頼

### Claude Codeに回す作業

- 文章構成や設計の見直し
- 丁寧なレビューや改善提案
- READMEや運用ルールの整理
- Codexの出力を別視点で確認したい場合

### Cursorに回す作業

- Cloud Agentとしてコード全体の確認が必要な場合
- 複数ファイルにまたがるコード修正の補助
- Cursorの開発環境やエディタ機能と組み合わせたい場合

### 節約ルール

- まずCodexに依頼する
- Codexで難しい、または不安が残る場合だけClaude CodeかCursorに依頼する
- 同じ作業を複数AIに同時依頼しない
- Claude CodeやCursorを使う前に「なぜCodexではなくこちらを使うのか」を一言メモする
- 大きな作業は小さく分けて、無駄な利用を避ける

### 共通ルール

- 変更前に対象リポジトリ、目的、触ってよいファイルを明記する
- 変更後は必ずdiffとPRを確認してからマージする
- 秘密情報、APIキー、個人情報、医療・介護の機微情報はSlackに書かない

## AI階層運用ルール

Slackでは、OpenClaw、Hermes Agent、Codex、Claude Code、Cursorを単独で使い切るのではなく、役割を分けてチームとして動かします。Slackを「AI指令室」として使い、相談、指示、判断材料、作業ログを同じ場所に残します。

基本は、実際のファイル編集やPR作成などの作業はCodexを優先します。Codexだけでは文章の精密レビューやローカル環境での細かい確認が足りない場合に限り、Claude CodeやCursorへ回します。最終判断、PRマージ、外部送信、納品、公開、重要な削除操作はAIだけで完結させず、必ず人間が確認します。

### 役割分担

| 担当 | 役割 | 主に任せること |
| --- | --- | --- |
| OpenClaw | 司令塔 | 次にやる作業、優先順位、担当AIを決める |
| Hermes Agent | 記録・監査係 | 会話ログ、決定事項、TODO、日報、抜け漏れ確認を整理する |
| Codex | 主力作業係 | GitHub上のREADME修正、Markdown整理、ファイル編集、コミット、PR作成を進める |
| Claude Code | 精密レビュー係 | 長文レビュー、文章の自然さ、構成整理、論理確認、複雑な意図理解が必要なときに使う |
| Cursor | 現場修正係 | ローカル編集、画面確認、細かい修正、ファイル単位の調整が必要なときに使う |
| 人間 | 最終承認者 | 成果物の採用、マージ、送信、公開、納品を判断する |

### AIループ運用の基本形

AI同士を回すときは、成果物の質を上げることが目的です。迷走、重複、コスト増、責任不明を避けるため、次の流れを基本形にします。

1. うーちゃんが依頼する
2. OpenClawが作業を分解し、優先順位と担当AIを決める
3. Hermes Agentが目的、条件、TODO、完了条件を整理する
4. Codexが実作業を行い、必要に応じてPRを作成する
5. Claude CodeまたはCodexがレビューする
6. Hermes Agentが決定事項、残課題、リスクを整理する
7. 人間が最終確認して、マージ・公開・納品を判断する
8. 必要なら今回の学びをREADMEや運用メモに反映する

### ループ回数の制限

- AI同士のレビュー・修正ループは最大3回までとする
- 3回で解決しない場合は、人間が確認する
- 同じ指摘が繰り返される場合は、無理に自動化せず人間判断に切り替える
- 完了条件が曖昧なままAIを回し続けない

### 合言葉ルール

Slackで短く指示したいときは、次の合言葉を使います。

| 合言葉 | 意味 |
| --- | --- |
| 次 | 現在の状況を整理し、次にやるべき作業を1つだけ提案する |
| レビュー | 直近の成果物を、目的・分かりやすさ・抜け漏れ・安全性の観点で確認する |
| 確定 | 今回決まったことを短くまとめ、必要ならREADMEや運用メモに反映する |

### レビュー観点

AIに成果物をレビューさせるときは、次の観点を確認します。

- 目的に合っているか
- 指示からズレていないか
- 読む人に分かりやすいか
- 重複や冗長さがないか
- 事実確認が必要な箇所はないか
- 個人情報・機密情報が混ざっていないか
- 公開・送信・マージしてよい状態か
- 次に人間が確認すべき点は何か

### 人間承認が必要な操作

次の操作は、AIの提案だけで進めず、必ず人間が確認してから実行します。

- GitHubのPRマージ
- 外部へのメール送信
- Slackでの重要投稿
- クライアント納品
- 個人情報を含むファイル処理
- 請求、契約、支払い関係
- 医療、介護、法務に関わる断定表現
- 重要ファイルの削除や大幅変更
